# Impact Format Reference

This document describes the JSON format of impact reports generated by `cargo xtask impact`.

## File Location

Impact reports are written to:

```
artifacts/impact/impact.report.v1.json
```

This is a fixed location that gets overwritten on each impact analysis run.

## Top-Level Structure

```json
{
  "changed_paths": [
    "path/to/changed/file.rs"
  ],
  "impacted_scenarios": [
    "SCN-XK-..."
  ]
}
```

## Fields

### `changed_paths`
- **Type:** `array` of `string`
- **Required:** Yes
- **Description:** List of file paths that were analyzed for impact. These paths are normalized (forward slashes, no leading `./`).

**Normalization Rules:**
- Backslashes (`\`) are converted to forward slashes (`/`)
- Leading `./` prefixes are removed
- This ensures consistent matching across Windows and Unix systems

### `impacted_scenarios`
- **Type:** `array` of `string`
- **Required:** Yes
- **Description:** List of scenario IDs that are potentially impacted by the changed paths. These scenarios should be re-tested to ensure the changes don't break anything.

## Impact Detection Logic

A scenario is included in `impacted_scenarios` if:

### 1. Allowed Edit Root Match

The changed path starts with one of the scenario's `allowed_edit_roots`:

```
Scenario allowed_edit_roots: ["crates/ixhtml-scan", "specs/features/sec"]
Changed path: "crates/ixhtml-scan/src/scanner.rs"
Result: IMPACTED (path starts with "crates/ixhtml-scan")
```

### 2. Crate Match

The changed path contains one of the scenario's crate names:

```
Scenario crates: ["efm-rules", "validation-run"]
Changed path: "crates/efm-rules/src/rules/inline.rs"
Result: IMPACTED (path contains "efm-rules")
```

## Complete Example

```json
{
  "changed_paths": [
    "specs/features/workflow/bundle.feature",
    "crates/xtask/src/main.rs"
  ],
  "impacted_scenarios": [
    "SCN-XK-WORKFLOW-001",
    "SCN-XK-WORKFLOW-002",
    "SCN-XK-WORKFLOW-003",
    "SCN-XK-WORKFLOW-004",
    "SCN-XK-WORKFLOW-005"
  ]
}
```

## Example Use Cases

### Parse Impacted Scenarios

```bash
# Get list of impacted scenarios as newline-separated values
cat artifacts/impact/impact.report.v1.json | jq -r '.impacted_scenarios[]'
```

Output:
```
SCN-XK-WORKFLOW-001
SCN-XK-WORKFLOW-002
SCN-XK-WORKFLOW-003
...
```

### Count Impacted Scenarios

```bash
cat artifacts/impact/impact.report.v1.json | jq '.impacted_scenarios | length'
```

### Check if Specific Scenario is Impacted

```bash
# Check if SCN-XK-WORKFLOW-001 is in the impacted list
cat artifacts/impact/impact.report.v1.json | jq '.impacted_scenarios | contains(["SCN-XK-WORKFLOW-001"])'
```

### Get Changed Paths

```bash
# List all analyzed paths
cat artifacts/impact/impact.report.v1.json | jq -r '.changed_paths[]'
```

## Empty Results

When no scenarios are impacted:

```json
{
  "changed_paths": [
    "some/file/that/touches/nothing.rs"
  ],
  "impacted_scenarios": []
}
```

This is valid output - it means the changes don't affect any tracked scenarios.

## JSON Schema

The impact format is defined in Rust at `crates/scenario-contract/src/lib.rs`:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize, Default)]
pub struct ImpactReport {
    #[serde(default)]
    pub changed_paths: Vec<String>,
    #[serde(default)]
    pub impacted_scenarios: Vec<String>,
}
```

## Interpreting Results

### Conservative Analysis

Impact analysis is intentionally conservative - it may report more scenarios than strictly necessary. This is safer than missing a potentially affected scenario.

### False Positives

You may see scenarios listed that don't actually need re-testing. This happens when:
- A crate is listed but the specific changed file doesn't affect the scenario's code path
- An `allowed_edit_root` is too broad

### False Negatives

The current implementation should not produce false negatives (missing impacted scenarios) as long as:
- The `allowed_edit_roots` are correctly configured in sidecar files
- The `crates` list accurately reflects dependencies
