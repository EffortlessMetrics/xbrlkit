## Problem Statement

Multiple crates manually construct `ValidationFinding` structs with similar patterns, resulting in code duplication and inconsistent field population:

| Location |
|----------|
| `crates/dimensional-rules/src/lib.rs` |
| `crates/numeric-rules/src/lib.rs` |
| `crates/efm-rules/src/lib.rs` |
| `crates/unit-rules/src/validator.rs` |
| `crates/numeric-rules/src/decimal_precision.rs` |

### Current Pattern (Duplicated)
```rust
ValidationFinding {
    rule_id: format!("PREFIX.{}", sanitize_for_rule_id(concept)),
    severity: "error".to_string(),
    message: format!("Concept '{}' ...", concept, value),
    member: Some(fact.member.clone()),
    subject: Some(fact.concept.clone()),
}
```

---

## Proposed Solution

Implement a **Builder Pattern** for `ValidationFinding` in the `xbrl_report_types` crate to provide:
- Fluent API for constructing findings
- Consistent field population
- Reduced boilerplate
- Type safety

### Target API Design
```rust
ValidationFinding::error("SEC.UNIT.001", "message")
    .with_member(&fact.concept)
    .with_subject(&fact.concept)
```
